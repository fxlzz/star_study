# 浏览器渲染原理

那么浏览器渲染原理呢，也是一个大重点，需要自己理解。
首先，浏览器上有很多进程，其中，网络进程和渲染进程（渲染主线程）都是我们要掌握的重点。

渲染主线程跟事件循环是紧密相关的，可以去看[[JavaScript/事件循环]]

---

一切的开始，就是网络进程获取了中的某个线程获取了 HTML 文件，并将其包装成任务，交到渲染主线程的**消息队列**。等待解析。

在渲染主线程中，步骤开始了

## ① 解析 HTML -> parse

这是第一步，将 HTML 文件转化成 DOM 树和 CSSOM 树，因为获取的 HTML 文件是”字符串“，不利于后续的操作，那么这一步就是将不好操作的字符串转化成更好操作的 JS 对象（DOM 树和 CSSOM 树）

### DOM 树

将 HTML 文件根据编码（如 UTF-8）转化成字符，在将字符转化成 Token（如`<html>、<body>`等标签）最后将 Token 转换成 DOM 树

#### DOM 操作的性能考虑

频繁的 DOM 操作可能导致性能问题，因为每次修改都可能触发重排（reflow）或重绘（repaint）。为提高性能，可以：

- 使用文档片段（DocumentFragment）批量操作 DOM。
- 离线操作 DOM（如先从 DOM 树上移除元素，修改后再插入）。
- 使用虚拟 DOM 技术（如 React 和 Vue 采用的方法）。

### CSSOM 树

#### CSSOM 对渲染性能的影响
CSSOM 的构建是**渲染阻塞**的，意味着在 CSSOM 完全构建之前，页面不会被渲染。为优化性能：

- 精简 CSS，移除未使用的样式。
- 使用媒体查询，使非关键 CSS 异步加载。
- 考虑内联关键 CSS。

### 注意：

1. 遇到`<script>`标签时，会暂停解析（除非使用`async`或`defer`属性），优先执行 JavaScript 代码，因为 JS 可能修改 DOM 结构。
2. 外部资源（如 CSS、图片）的加载**不会阻塞 HTML 解析**，但可能影响后续渲染步骤。渲染主线程在解析 HTML 时，遇到`<link src="...">`，不会停止等待 CSS 解析完成，会继续往前走，不会阻塞。它会先开启一个**预解析线程**来优先下载 CSS 和 JS（这个毕竟是外联的嘛）这个**预解析线程**，会将 CSS 解析成。更有利于渲染主线程解析生成 CSSOM 树的结构。

所以，这也就是为什么 CSS 不会造成页面卡死，因为它解析都不在渲染主线程上。

但是，对于 JS 代码是在渲染主线程上执行的。

## ② 计算样式 -> style

这一步，是将 CSS 的最终样式，计算出来，结果是生成一颗带有最终样式的 DOM 树。

跟用 `getComputedStyle` 的结果是一样的

这部分，会将两部分的样式根据一定的策略合并在一起，

- 浏览器默认样式表
- 作者样式表

- 内部样式 `<style>...</style>`
- 外联样式`<link rel="stylesheet" href="">`
- 内联样式`<div style=""></div>`

## ③ 生成布局树 -> layout

这一步会生成一颗 布局树（layout tree），会将一些没有几何信息的元素，从 dom 树上去掉。

所以，DOM 树和 Layout 树并非一一对应。

比如，如果某个块盒元素设置了（`display: none`）那么它在局部树就不会在节点数据，因为它没有几何信息。在比如，`div::before { content: "" }` 的话，那么这个`::before` 虽然在 DOM 树上不可见，但是在 Layout 树上会有节点。

## ④ 分层 -> layer

这一步，会根据布局树，将页面分成几个层次。如果某个层次，操作很多，那么只需要重新渲染那一个页面，不用修改其他的层次。这样会提升页面渲染速度。

这个分层，开发者是不能修改的，只能修改某些属性稍微影响浏览器分层的决策。

其实，就是哪些会影响**堆叠上下文**的属性，

例如，`z-index、transform、will-change、opacity`

```
div {
  will-change: transform; // 表示这个div可能因为transform多次修改，希望单开一个层次
}
```

## ⑤ 绘制 -> paint

这一步，会将每一层都转化成 **绘制指令，**像操作 canvas 就类似于这个绘制指令

什么是绘制指令呢？

举个简单例子，

1. 将鼠标移动到 x: 200 y: 100 的位置
2. 画一个长 200 宽 200 的矩形
3. 将矩形用红色填满

就类似于上面的这种，每一行都是一个绘制指令

---

上面这个**五**个步骤，都发生在渲染主线程上，

下面的步骤，发生在**合成线程**（是渲染进程中的一个线程）

---

## ⑥ 分块 -> Tiling

接收到渲染主线程的指令集，将每一层划分成**小小的块区域**

类似于下面的，分成一格一格的，

![](https://cdn.nlark.com/yuque/0/2025/png/51701855/1751289534109-688dae4c-6937-4097-9401-67484c825325.png)

## ⑦ 光栅化 -> raster

将每块变成位图，位图就是算出了每一个位置的**像素，**这个步骤会用到 **GPU 加速**

这个会优先将离**视口**的位置，先**光栅化**

## ⑧ 画 -> draw

根据每个像素点，画页面，这个地方还会计算一些 css 属性，比如，transform（设置这个属性，不会引起重排和重绘）

这个部分，合成线程就将东西都交给 GPU 进程，然后 GPU 交给显卡（调用操作系统的接口）

这里有一个小知识点，渲染进程是放在**沙盒**中，与计算机硬件是隔离开的 ---> 安全

# 什么是 reflow？

这是网页性能优化的范畴，reflow（回流，重排），它是指当 DOM 的变化影响到了元素的布局信息（元素的大小尺寸、边距、位置、浏览器窗口的大小）

也就是**修改元素的几何信息**，那么会浏览器会重新生成 CSSOM ，重新生成后面的布局树，剩下的都要重新渲染一遍。

## 触发 reflow 的场景

1. DOM 的添加、删除、修改
2. 尺寸属性的修改：改变元素的 width、height、margin、padding 登
3. 位置相关：定位、浮动
4. 布局相关：当调用`offsetWidth`、`offsetHeight`、`getBoundingClientRect()`等方法**获取元素的布局信息时**，浏览器为了返回准确的数据，会**强制触发**一次 reflow。
5. 浏览器窗口尺寸改变

## 怎么减少 reflow 呢？

1. 批量修改 dom 元素：可以使用`DocumentFragment`批量操作 dom 元素，这样只需要一次 reflow

```js
const list = document.querySelector("#list");
const fruits = ["Apple", "Orange", "Banana", "Melon"];
const fragment = new DocumentFragment();
fruits.forEach((fruit) => {
  const li = document.createElement("li");
  li.textContent = fruit;
  fragment.appendChild(li);
});
list.appendChild(fragment); // 只有这里会触发一次reflow
```

2. 使用 `transform`因为这个属性，在最后一步”画“，不会触发 reflow
3. 离线操作 dom 元素，常见的操作方式

“离线操作 DOM” 指的是在 DOM 树之外对元素进行操作，之后再将其添加到文档中，或者暂时隐藏元素（使其脱离渲染流程），完成修改后再重新显示。

1. 使用`DocumentFragment`
2. 先删除元素（一次 reflow），在操作元素，随后在添加（一次 reflow）或先隐藏元素，修改完之后，在添加。这两种方法都只会产生两次 reflow

```js
const element = document.getElementById('target');

// 方式一：移除元素  ---> 产生一次reflow
const parent = element.parentNode;
parent.removeChild(element);

// 进行多次修改  ---> 这些修改都不会 reflow
element.style.width = '200px';
element.style.height = '300px';
element.textContent = 'New Content';

// 修改完成后重新添加  ---> 产生一次reflow
parent.appendChild(element);

// 方式二：设置display: none
element.style.display = 'none';
// 多次修改...
element.style.display = 'block';
```

# 什么是 repaint？

重绘，指的是元素的外观发生改变，与元素的几何信息无关。

## 触发 repaint 的场景

1. 修改颜色：color、background-color、border-color
2. 是否可见：visibility、opacity
3. 阴影效果：box-shadow、text-shadow
4. reflow 一定会触发 repaint

# URI 与 URL

URI：统一资源标志符，URL：统一资源定位符

注意：这是两个概念，URL 是 URI 的子集

举个例子，

人 使用 身份证 来唯一标志，人的身份。这个唯一标志（即身份证）就是 URI，那么 URL 是什么呢？其实可以将 URL 理解成为标识这个人的地址，这个地址，每个人也是唯一的。实现了 URI 的功能（体现了 URL 是 URI 的子集）

  

比方说，

有这么一个人，用 URL 来描述：

动物地址协议://地球/中国/xx 省/xx 市/xx 县/xx 村/xx 组 xx 号

那么，用 URI 来描述：

身份证：123456789

所以不论是用定位的方式还是用编号的方式，我们都可以唯一确定一个人，都是URl的一种实现，而URL就是用定位的方式实现的URI。
