## Improving_coding 项目的来源？
为什么会做这个项目呢？
其实来源于上家实习公司，业务是做 B 端数据库相关的东西。有几个问题：
1. 代码大量重复（因为每个数据库都有相似的功能，但也有其特殊的部分）其实大部分逻辑都差不多
2. B 端客户定制化需求——每一个客户都配置一条新的分支，代码重复并且不好管理

---
项目架构简要说明：
使用 nodejs 搭建 bff 层，作为运行时服务器，处理 API 响应和页面响应。页面使用 DSL 描述 UI，每一套页面都是独立的 Vue SPA 项目（使用的组件来源于公共组件库）。


## 基于 Vue 3+Element UI 封装通用组件库（列表 / 表单等）、状态管理库及工具库，实现组件复用率 80%+

问：你是怎么实现组件复用率有 80%的？
答：在项目架构上，页面 UI 是由 DSL 来描述的。因为项目来源于 B 端设计，大部分都是相同的 CURD 场景。因此，将 80%的标准场景抽象为“配置驱动”。这样做，可以让开发者从维护系统代码转变为维护 DSL 配置文件。
结果上，页面的 80%逻辑由通用组件承载，剩下 20%允许定制化开发（留有插槽）。

问：你是怎么设计一个通用的“动态表单”组件的吗？
答：设计思路分为两步：
+ 描述： 定义一套标准的 JSON 协议，包含字段名 key、类型 type、校验规则 rules、联动逻辑以及布局属性等。

+ 解析渲染： 利用 v-for 循环解析 schema，通过映射动态绑定组件，如：type='input' 映射到 el-input 组件。

+ 提升功能： schema 中支持表达式或函数字符串。例如，当 type 为 A 时，隐藏字段 B。在内部维护了一个观察者，当特定字段变化时重新计算 schema 的状态


问：你的组件是如何保证灵活度和低耦合的？
答：支持属性穿透 `v-bind`，将所有未定义的 props 直接穿透给底层的 element ui 组件。支持用户在 DSL 中自定义属性以及保留 element ui 的原有属性

插槽优先，能用插槽解决就用插槽解决，不增加 props。组件本身只负责“什么时候显示”和“数据怎么流转”

代码解耦合，UI 组件尽量减少具体 API 请求，通过 hook 获取具体数据，UI 组件仅负责渲染数据，不负责获取数据（react query 修改）

## 开发通用中间件 ：实现接口参数校验、签名校验、全局错误捕获、请求日志记录，保障接口调用的安全性和可追溯性；
问： 你的签名校验具体是如何实现的？
答： 为了增强后端服务，因此对 API 接口设计了签名校验。具体实现，采用对称加密思想。
利用三个参数（时间戳、随机串、密钥）进行 md 5 加密，客户端每次请求接口时携带（时间戳、随机串、签名），服务端利用服务端密钥进行加密，判断签名是否一致。
签名一致，给予响应，否则，返回错误状态码。

问： 全局错误捕获中间件是怎么实现的？
答： 后端使用 koa 框架，中间件模型为洋葱圈模型，全局捕获中间件位于最外层。设计思路，内部代码有错误统一向上抛出 throw，传递错误信息和 code。所有异常会被中间件捕获，统一封装为后端响应结构，返回给前端。


## 完成 Node. Js 服务的路由分发、Controller 业务逻辑处理、Service 层接口封装，对接 MySQL 数据库实现数据的增删改查与持久化

问： 后端服务架构是如何设计的？
答： 后端采用经典的架构模式，路由层、controller 层以及 servic 层，各层各司其职，降低代码耦合度。
路由层： 负责 API 请求转发以及页面请求转发至对应 controller
Controller 层： 负责调用 service 层获取数据，实现业务逻辑，返回响应。
Service 层： 原子化对数据的增删改查操作，为 controller 层提供数据服务。

## 分析构建和打包结果，优化webpack分包策略，分离非首屏代码，实现懒加载。替换js解析loader，添加webpack缓存策略，将构建时长从30s左右降低至9s

前提：
构建代码 -> 40MB -> 本地开发需 30-32s
打包代码 -> 6.8MB -> 首屏加载时间特别长

从两方面进行优化：
- 构建
- 传输

*构建* 
- js 解析 loader 慢 -> swc-loader
- loader 的解析范围不明确 -> include， 补充 exclude /node_modules/
- 没有缓存 -> 配置 webpack 持久化缓存 cache 
- 采用多进程 -> thread-loader

*传输*
根本原因： webpack optimization 配置项 -> splitChunk 没有配置 -> index.js 这个文件包含所有的依赖项，包特别大
- 进行详细的分包 
	- react 核心依赖包
	- 状态管理库
	- ui 组件库
	- echarts 图表库
	- 自定义库
	- 其他第三方依赖
	- common 库
---

解决：
- 分析那些是首屏的代码，那些不是
	- 分离 -> 进行懒加载 -> 独立的chunk
